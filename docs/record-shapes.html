Record Shapes

final JSON for every chain visibility

Public Light Energy Dark Matter · Locked Dark Matter · Unlocked Anonymous Dark Matter

The same record, the same image, the same constellation — represented in each chain visibility. Identifier, content hash, and signature are identical across all forms (the bar carries identifier + content_hash; the signature covers both). What changes is which fields are plaintext, which are sealed, and whether the structure of the record itself is visible.

Every dark_matter record has the same top-level shape. No sentinel values, no leaked field set. The only way to learn what a dark_matter record contains is to unlock it.

Public Chain no chain_visibility

Default. No password ever set. Every field plaintext. GPS rides the time-lock puzzle inside gps_time_locked — anyone can recover it in ~10 years via sequential squaring; nobody can shortcut without the trapdoor (already discarded at mint time).

Dark Matter · Locked chain_visibility: "dark_matter"

Full opacity. The protected fields and the chunks namespace are deleted from the record, not replaced with sentinel values. Every dark_matter record has the identical top-level shape — no per-record field-set leak. A LoRA-using record looks the same as a non-LoRA one. An epagomenal-day record looks the same as a regular Tuesday.

Public — required for system function

  • identifier, content_hash, hash_version
  • signature, public_key, key_fingerprint
  • creator_name
  • parent_id
  • constellation_name, heart_star_id, constellation_index
  • decoder_hash (cross-Age constant)
  • chain_visibility: "dark_matter"

Sealed (deleted from record + envelope-only)

  • − prompt, seed, generation params
  • − birth (celestial + machine)
  • − birth_traits
  • − rarity, rarity_score
  • − thumbnail
  • − constellation_hash
  • − chunks (the entire namespace)
  • + encrypted_fields (envelope)
  • + encrypted_chunks (envelope)
  • + gps_password_locked (envelope)

Dark Matter · Unlocked in-memory only

The same record after the viewer enters the password. The browser runs PBKDF2 + AES-GCM via SubtleCrypto, decrypts the three envelopes, parses each JSON blob, substitutes the fields back into the record. The result is rendered, then forgotten. Anonymous Dark Matter optional knob

A creator who wants their chain anonymous (not just sealed but unattributed) can move creator_name into PROTECTED_FIELDS, so it ends up inside encrypted_fields. Viewers see only the key_fingerprint until unlock — the chain is still identifiable by key (signatures need it), but no human name is attached.

Public

  • identifier, content_hash, hash_version
  • signature, public_key, key_fingerprint
  • − creator_name (moved into encrypted_fields)
  • parent_id, constellation_*
  • decoder_hash, chain_visibility

Sealed

  • everything from dark_matter
  • + creator_name (now inside encrypted_fields)

  "content_hash": "8ec3f6dd1c47d4d7",
  "hash_version": 1,

  "public_key": "86cb4ed6af3fd6c5...",
  "key_fingerprint": "86cb:4ed6:af3f:d6c5",
  // no creator_name — it lives in encrypted_fields
  "parent_id": "mememage-prev123abc",
  "constellation_name": "Gilamul",
  "heart_star_id": "mememage-genesis001",
  "constellation_index": "ε",
  "decoder_hash": "5e84a3b9c601d4d7",
  "chain_visibility": "dark_matter",

  "encrypted_fields":   { /* now also seals creator_name */ },
  "encrypted_chunks": { /* envelope */ },
  "gps_password_locked":    { /* envelope */ }
}

key_fingerprint can never be sealed — it's how the signature verifies, so it's always public. A persistent observer can still see "a chain by 86cb:4ed6 made these records", but cannot connect that fingerprint to a human name without either out-of-band knowledge or unlocking a record.

This is a knob, not a default. The system ships with creator_name public for TOFU naming. Creators who want anonymity opt in by extending PROTECTED_FIELDS.

The Envelope internal structure

Every *_encrypted field has the same four-key shape. AES-256-GCM with a key derived via PBKDF2-HMAC-SHA256 (600,000 iterations). Per-record salt blocks rainbow tables; per-field IV ensures structural distinctness even when two records seal identical plaintext under the same password. 

{
  "salt": "<16 bytes hex>",    // PBKDF2 salt — fresh per record
  "iv":   "<12 bytes hex>",    // AES-GCM nonce — fresh per encryption
  "ct":   "<N bytes hex>",     // AES-256-GCM ciphertext (no tag appended)
  "tag":  "<16 bytes hex>"     // GCM authentication tag, separate field
}

Wrong password → GCM tag fails → InvalidTag exception → Wrong password in the UI. No partial decryption. No oracle. No timing leak from a constant-time comparison.
